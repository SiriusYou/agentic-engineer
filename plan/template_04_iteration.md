# Template 04 — 反馈修正（Iteration Loop）
# 用途：将压力测试发现的漏洞反馈给 AI，修订 SDD
# 使用对话：回到写 SDD 的那个 Gemini 对话（Phase 2 的对话）
# 新建对话：不需要，复用 Phase 2 的对话

---

## 使用步骤

1. 回到 Phase 2 的 Gemini 对话（还有 SDD 的上下文）
2. 粘贴以下 Prompt，将漏洞列表填入
3. 输出结果保存为 `spec_v2.md`（版本号递增）
4. 重新执行 Template 03（新建对话，重测）
5. 循环直到满足收敛阈值（0 高严重度 + ≤3 中严重度）

---

## Prompt

```
压力测试发现了以下设计问题，请针对每个问题修订 SDD 对应章节。

发现的问题：
[粘贴漏洞记录表中"⚠️有"的条目，格式如下]

问题1（来自 W2，高严重度）：
当前设计没有在数据层面检查资源归属，只在业务层检查，
攻击者可以绕过业务层直接调用数据访问函数。

问题2（来自 U5，中严重度）：
第三方支付服务不可用时，系统没有定义降级策略，
用户会看到无响应而非明确的错误提示。

[继续列出其他问题...]

---

修订要求：
1. 只修改有问题的章节，不改动其他部分
2. 每个修订点说明"原来是什么"和"改成什么"
3. 修订完成后输出完整的新版 SDD
```

---

## 收敛判断

| 情况 | 行动 |
|------|------|
| 高严重度问题 > 0 | 必须修订后重测，不可跳过 |
| 中严重度问题 > 3 | 建议修订后重测 |
| 中严重度问题 ≤ 3，低严重度 | 可选：修订后执行，或记录为已知 debt |
| 0 高 + 0 中（全部通过）| 锁定 spec，进入执行 |

---

## 版本管理

每次修订递增版本号：

```
项目目录/
└── spec/
    ├── raw_requirements.md
    ├── spec_v1.md
    ├── scorecard_v1.json       （结构化评判数据）
    ├── stress_test_v1.md
    ├── spec_v2.md              （修订后）
    ├── scorecard_v2.json       （重测评判数据）
    ├── stress_test_v2.md       （重测记录）
    ├── spec_v3.md              （如需再次修订）
    └── spec_final.md           （最终锁定版本，复制自最后一个 vN）
```

**`spec_final.md` 一旦创建，不再修改。**
这是交给 Claude Code 执行的唯一输入。

---

## 移交执行

当 `spec_final.md` 锁定后，在 Claude Code 中输入：

```
请按照 spec-driven-dev 执行规范，实现 spec/spec_final.md 中定义的系统。
原子化提交，严格对应 Spec 章节，发现歧义立即停下来报告。
```
