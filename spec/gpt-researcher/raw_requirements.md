# Raw Requirements — Output Alignment Scoring + Auto-Retry

> TRACK-005 MVP Feature for gpt-researcher
> Date: 2026-03-05
> Source: Codebase exploration of ~/dev/gpt-researcher

---

## 1. 核心目标（这个功能要解决什么问题）

gpt-researcher 当前能生成研究报告，但没有机制评估"报告内容是否真正回答了用户的原始查询"。用户只能人工判断报告质量。本功能在研究完成后自动评估输出与 prompt 的对齐程度，不满足阈值时自动重试改进，减少人工审查负担。

## 2. 目标用户（谁会用它）

- **gpt-researcher 的 API 调用者**：通过 `GPTResearcher` 类编程调用的开发者，需要可靠的输出质量保证
- **CLI 用户**：通过 `cli.py` 运行研究任务，希望得到与查询高度相关的报告
- **自动化管道运营者**：在无人值守管道中运行研究任务，需要质量闸门避免低质量输出进入下游

## 3. 核心功能列表

- 用户可以在研究完成后获得一个 0-10 的对齐评分，表示报告内容与原始查询的匹配程度
- 用户可以设置对齐阈值（默认 7.0），低于阈值时系统自动重新研究
- 用户可以设置最大重试次数（默认 2），防止无限循环和成本失控
- 用户可以在报告元数据中看到最终对齐评分、重试次数、每轮评分历史
- 用户可以通过配置禁用自动重试，仅获取评分（advisory 模式）
- 用户可以通过 WebSocket 实时看到"正在评估对齐度..."的状态更新

## 4. 已知约束

- **技术栈**：必须用 Python，集成进现有 `gpt_researcher/` 包结构
- **LLM 依赖**：对齐评分本身需要 LLM 调用，会增加 API 成本（每次评估约 1 次 LLM 调用）
- **现有架构**：`GPTResearcher` 类已有 `conduct_research()` → `write_report()` 流程，评分需在 `write_report()` 之后插入
- **配置系统**：必须通过现有 `Config` 类（`gpt_researcher/config/`）管理新参数
- **WebSocket 协议**：实时状态更新必须兼容现有 `websocket_manager.py` 的消息格式
- **性能**：单次评分应在 10 秒内完成；含重试的完整流程不应超过原始研究时间的 3 倍

## 5. 已知风险

- **评分可靠性**：LLM 对自身输出的评估可能存在偏差（自我评价倾向偏高）；需考虑是否用不同模型/prompt 评估
- **成本放大**：自动重试会成倍增加 API 和搜索成本；需要严格的重试上限和成本追踪
- **重试收敛性**：重复研究可能返回相似结果，导致评分无法提升；需要检测"评分停滞"并提前终止
- **评分维度单一**：仅用"对齐度"一个维度可能不够（完整性、准确性、深度也重要）；MVP 先做单维度，后续可扩展

## 6. 明确不做的事

- 不做多维度评分（准确性、深度、格式等）— MVP 仅做 prompt 对齐度
- 不做人工反馈闭环（如用户标记"这个报告不好"触发重试）
- 不做评分模型微调或专用评分模型训练
- 不做前端 UI 展示评分（仅 API/CLI 层面输出）
- 不改变现有 `SourceCurator` 的源评估逻辑
- 不做跨 report_type 的差异化评分标准（所有类型用同一评分 prompt）

---

## MVP 验收阈值（TRACK-005 Scope Lock）

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 对齐评分阈值 | 7.0 / 10 | 低于此值触发重试 |
| 最大重试次数 | 2 | 含首次共最多 3 轮 |
| 可接受耗时上限 | 原始研究耗时 × 3 | 超时则返回最佳结果 |
| 评分停滞检测 | 连续 2 轮评分提升 < 0.5 | 提前终止重试 |
