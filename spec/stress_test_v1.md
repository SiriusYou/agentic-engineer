## 漏洞详情

### C1 [HIGH] FileRefChecker 误报风险

**问题：** 路径提取规则第 2 条（"代码块中引用真实文件的路径"）无法区分两种目录树代码块：
1. **项目结构描述**（如 README.md 中展示实际目录结构）→ 应检查路径是否存在
2. **预期输出描述**（如模板中展示 `spec/` 将来会有的文件）→ 不应检查，因为文件尚不存在

例如 `template_03_stress_test.md` 中的 `scorecard_v1.json` 在项目初始状态下不存在，但它是"预期产物"而非"断裂引用"。

**修复建议：** 定义代码块上下文分类规则，或限制目录树检查仅适用于 README.md。

### U8 [MEDIUM] 空状态行为未定义

**问题：** 每个检查器的 `run()` 方法在目标数据不存在时应返回什么未定义：
- `SpecNamingChecker`：spec/ 目录不存在或为空 → 返回空列表？还是一个 WARNING？
- `TrackStatusChecker`：tracks.md 中无 track 条目 → 返回空列表？还是 PASS？
- `QuestionIdChecker`：stress-test-prompts.md 不存在 → WARNING 还是 FAIL？

**修复建议：** 为每个检查器定义"无数据"场景的返回值。

### U10 [MEDIUM] 诊断信息不足

**问题：** `CheckResult.message` 字段描述为"人类可读的检查结果描述"，但未要求包含实际提取值。例如 ConvergenceChecker 发现不一致时，应报告"plan/quick_reference.md:16 期望阈值 3，实际提取 5"，而非仅说"收敛阈值不一致"。

**修复建议：** 在 CheckResult 或 message 格式中要求包含 expected/actual 值。

### C2 [MEDIUM] 收敛阈值匹配策略模糊

**问题：** 匹配策略仅描述为"搜索包含'高'和'中'的阈值描述行，提取数字"。未给出：
- 具体正则表达式
- 如何处理多个数字在同一行的情况
- 文字数字（"零"、"三"）是否纳入匹配范围

**修复建议：** 给出每个文件的具体提取正则，限定为阿拉伯数字匹配。

### C3 [MEDIUM] tracks.md 表格解析策略

**问题：** 解析逻辑仅说"解析表格行，提取 Status 列的值"，未定义：
- 按列位置（第 1 列）还是按列标题（"Status"）匹配
- 分隔符处理（前后空格、管道符转义）
- 表格不存在或格式异常时的回退

**修复建议：** 指定按列标题匹配，定义具体的行分割和清理逻辑。
