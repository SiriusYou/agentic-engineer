# 规划阶段工作流（Planning Workflow）

> 这是 Execution Agent 之前的人工规划阶段文档。
> Claude Code Execution 不直接执行这些步骤，但可以在用户提问时参考解释。

---

## 阶段总览

```
人类 (Human)          Planning AI             Execution Agent
    │                      │                        │
    ▼                      │                        │
灵感捕获                    │                        │
```

## 阶段 1：灵感捕获（Human）

**时长：** 20-60 分钟

**工具：** WisprFlow 或任意语音/文字工具

**做什么：**
- 自由倾倒所有想法，不做自我审查
- 覆盖：功能需求、用户故事、技术直觉、边界条件、已知风险
- 不需要结构化，越原始越好

**输出：** 原始需求文本（500-2000字）

**注意：** 这是唯一需要人类深度参与的创意阶段，质量决定后续所有输出。

---

## 阶段 2：AI 结构化（Planning AI Round 1）

**工具：** 新建 Gemini 对话

**输入 Prompt 模板：**
```
请根据以下需求描述，生成一份完整的软件设计文档（SDD）。

需求描述：
[粘贴原始需求文本]

SDD 需要包含以下部分：
1. 项目概述（目标、核心用户场景、系统边界）
2. 技术栈选型（含选择理由）
3. 系统架构（组件划分、职责、依赖关系）
4. 接口定义（所有 API 端点，含请求/响应格式）
5. 数据模型（实体定义、字段类型、索引策略）
6. 错误处理策略

请确保每个部分足够详细，可以直接交给开发工程师实现，无需进一步沟通。
```

**输出：** `spec_v1.md`

---

## 阶段 3：对抗性压力测试（Planning AI Round 2）

**工具：** 全新 Gemini 对话（必须新开，不能复用上一个）

**步骤：**
1. 粘贴完整 `spec_v1.md`
2. 按项目类型选取对应层级的问题（详见 `stress-test-prompts.md` 中的项目类型选择表）
3. 记录每个发现的漏洞，收集 JSON scorecard

**判断：**
- 发现漏洞 → 进入阶段 4
- 无重大问题 → 进入收敛

---

## 阶段 4：反馈修正（Iteration Loop）

**触发：** 压力测试发现问题

**操作：**
1. 将所有发现的问题整理成列表
2. 反馈给 Planning AI（可复用 Round 1 的对话），要求修订对应章节
3. 输出 `spec_v2.md`
4. 重新进行压力测试（用新对话）
5. 循环直至满足收敛阈值

**退出标准：** 收敛阈值: 0 高严重度 + ≤3 中严重度

**输出：** `spec_final.md`（锁定，不再修改）

---

## 阶段 5：移交执行

将 `spec_final.md` 交给 Claude Code，并说明：
```
请按照 spec-driven-dev 执行规范，实现 spec_final.md 中定义的系统。
原子化提交，严格对应 Spec 章节，发现歧义立即报告。
```

---

## Phase 0：项目复盘（推荐）

项目完成后，使用 `plan/template_00_postmortem.md` 执行复盘：

- 记录每步耗时、迭代轮次、严重度下降曲线
- 评估规划流程有效性
- 收集改进数据，驱动后续模板迭代

> Phase 0 编号为 00，表示对整个 Step 1-5 流程本身的回顾。推荐每个项目完成后执行。

---

## 变更请求流程（Change Request）

`spec_final.md` 锁定后发现需要修改时，走 CR 流程而非直接改动。
完整的 CR 流程定义请参阅 `plan/quick_reference.md` 中的「变更请求流程」章节。

**核心原则：**
- 绝不修改已锁定的 `spec_final.md`
- 复制为 `spec_final_v2.md` 后只改必须改的章节
- 对新版本重新执行 Step 3 压力测试
- 满足收敛阈值（0 高 + ≤3 中）后锁定新版本
