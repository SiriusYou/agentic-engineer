## 4. 接口定义 [Required]

> 根据项目类型选择 4.A（Web/API）或 4.B（CLI 工具）。两种变体不可混用。

### 变体 4.A：Web/API 接口（Web 应用、API 服务使用）

#### 4.A.1 外部 API

每个端点：
```
METHOD /path/to/endpoint
描述: 这个接口做什么

请求:
  Headers: { Authorization: Bearer <token> }
  Body: {
    field_name: type  // 描述，是否必填
  }

响应:
  200: { ... }
  400: { error: string }
  401: { error: "Unauthorized" }

副作用: 会触发什么（发邮件、写数据库等）
```

#### 4.A.2 内部接口/函数签名

```typescript
// 函数名(参数: 类型): 返回类型
// 描述：做什么，什么情况下抛出异常
```

### 变体 4.B：CLI 接口（命令行工具使用）

#### 4.B.1 命令行参数

```
用法: tool_name [选项] <位置参数>

位置参数:
  input_path          输入文件或目录路径

选项:
  --flag VALUE        描述（默认值：X）
  --verbose           启用详细输出
  -o, --output FILE   输出文件路径（默认: stdout）

环境变量:
  TOOL_ROOT           项目根目录覆盖（默认: 自动检测）
```

#### 4.B.2 stdin/stdout/stderr 契约

| 通道 | 用途 | 格式 |
|------|------|------|
| stdin | 管道输入（可选） | 与位置参数互斥，格式一致 |
| stdout | 主输出（报告、数据） | 人类可读（默认）或 JSON（`--format json`） |
| stderr | 诊断信息、错误、进度 | 自由格式，不影响管道下游 |

#### 4.B.3 Exit Code 语义

| 退出码 | 含义 | 触发条件 |
|--------|------|---------|
| 0 | 成功 | 所有检查通过 / 正常完成 |
| 1 | 验证失败 | 输入合法但检查发现问题 |
| 2 | 使用错误 | 参数缺失、文件不存在、格式错误 |

#### 4.B.4 POSIX 组合性

描述工具与管道、重定向的交互方式：
- 支持 `tool input.md | grep FAIL` 管道链
- 支持 `tool input.md > report.md` 输出重定向
- stderr 不干扰管道下游（诊断信息仅写 stderr）

## 8. 模式定义约束 [Required for tools with pattern matching]

> 适用于包含正则表达式、glob 模式或其他匹配器的工具（如 lint 工具、检查器）。

### 约束规则

1. **精确正则**：所有检查器/匹配器的匹配逻辑必须给出精确的正则表达式，禁止使用"或等价表述""或类似模式"等模糊措辞
2. **正反例验证**：每个正则必须附带至少 3 个正例（应匹配）和 3 个反例（不应匹配）
3. **边界说明**：注明已知的误报/漏报边界（如"不处理代码块内的路径"）

### 模板

```
检查器: [名称]
匹配模式: `正则表达式`
标志: re.MULTILINE / re.VERBOSE / ...

正例（应匹配）:
  1. [输入] → [期望匹配结果]
  2. [输入] → [期望匹配结果]
  3. [输入] → [期望匹配结果]

反例（不应匹配）:
  1. [输入] → 不匹配
  2. [输入] → 不匹配
  3. [输入] → 不匹配

已知边界（误报/漏报）:
  - 误报: [场景描述] → [为什么会误报] → [缓解方案或标记为 accepted]
  - 漏报: [场景描述] → [为什么会漏报] → [缓解方案或标记为 accepted]
  - [其他边界]: [行为说明]
```

> **标准化要求（来自 TRACK-003 postmortem）：** 每个检查器必须包含"已知边界"段，明确列出已知的误报和漏报场景。此段的目的是让使用者在运行工具前了解预期的检测盲区，避免对工具能力产生误判。如果某个检查器没有已知的误报/漏报，应显式写明"无已知边界"。

## Spec 完整性检查

Execution Agent 在开始前验证：
- [ ] 章节 1-6 全部存在
- [ ] 每个组件都有清晰的职责描述
- [ ] 每个 API 端点有请求/响应格式（4.A）或命令行参数/exit code 定义（4.B）
- [ ] 所有环境变量都已列出
- [ ] 没有"待定"或"TBD"的关键决策
- [ ] 包含模式匹配的工具：章节 8 存在，每个正则有 3 正例 + 3 反例
- [ ] 无"或等价表述""或类似模式"等模糊措辞
