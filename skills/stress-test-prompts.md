# 对抗性压力测试 Prompt 库

> 用途：在 Spec 完成后，用全新的 AI 对话（Gemini 新窗口）逐一提问，
> 找出设计漏洞。发现问题则回到 Planning AI 修订 Spec，直至通过收敛阈值。

**使用方法：**
1. 新建一个全新 AI 对话（不能复用写 SDD 的对话）
2. 先粘贴完整 SDD，再根据项目类型选择对应的问题集
3. 记录每个发现的问题，汇总后一次性反馈给 Planning AI 修订

---

## 项目类型选择

根据项目特征选择问题集组合：

| 项目类型 | 使用的问题集 | 总题数 |
|---------|------------|--------|
| 无状态 CLI 工具（lint、格式化、检查器） | CLI 专用层 | 5 |
| 有状态 CLI / 纯逻辑库 | 通用层 | 10 |
| Web 应用 / API 服务 | 通用层 + Web/API 层 | 15 |
| 数据密集型系统 | 通用层 + 数据层 | 15 |
| 全栈 Web + 数据系统 | 通用层 + Web/API 层 + 数据层 | 20 |

> **CLI 工具判定规则：** 如果项目满足以下全部条件，使用 CLI 专用层而非通用层：
> - 无持久化状态（不写数据库、不维护会话）
> - 无网络 I/O（不调用外部 API、不启动服务器）
> - 输入来源仅为文件系统和 stdin
> - 输出通过 stdout/stderr + exit code

## 〇、CLI 专用层（无状态 CLI 工具，5 题）

这些问题专为无状态命令行工具设计（lint 工具、格式化器、检查器等）。
替代通用层 U1-U10 中对无状态 CLI 不适用的题目。

**C1. 输入边界**
> 工具收到以下输入时的行为是否被明确定义？(a) 空文件（0 字节）(b) 超大文件（100MB+）(c) 非 UTF-8 编码文件（如 Latin-1、二进制文件）(d) 包含 BOM 头的 UTF-8 文件。哪些情况应报错退出，哪些应优雅跳过？

**C2. 正则/匹配器鲁棒性**
> 工具中每个正则表达式或匹配器的误报和漏报边界在哪里？请给出一个合理的输入，使正则产生意外的匹配（误报）或遗漏一个应匹配的内容（漏报）。

**C3. 空目录 / 无匹配文件**
> 如果目标目录为空、或所有文件都不满足 glob/过滤条件，工具是静默返回成功（exit 0），还是报告"未找到文件"警告？这两种行为哪种正确？用户如何区分"全部通过"和"什么都没检查"？

**C4. 路径解析歧义**
> 以下路径场景是否都有明确行为？(a) 相对路径 vs 绝对路径 (b) 符号链接指向的目标 (c) 路径中包含空格或特殊字符 (d) `--root` 参数与 cwd 不一致时的优先级。

**C5. 输出格式与 Exit Code 正确性**
> stdout 输出是否能被管道下游正确解析（如 `tool | grep FAIL`）？stderr 是否会污染 stdout？exit code 是否严格区分"检查通过"（0）、"检查发现问题"（1）和"工具本身出错"（2）？

## 一、通用层（所有项目必问，10 题）

这些问题测试任何软件系统都必须具备的基本健壮性。

**U1. 资源泄漏**
> 如果某个用户的操作在中途被网络中断，系统是否会产生孤立的资源（未关闭的连接、未释放的锁、半完成的事务）？

**U2. 并发写入冲突**
> 两个用户同时修改同一条 [实体] 数据，谁的修改会胜出？当前设计是否有乐观锁或悲观锁机制？

**U3. 分布式事务边界**
> [操作A] 和 [操作B] 需要原子完成，但它们跨越了两个不同的模块/服务。如果 A 成功但 B 失败，数据会处于什么状态？如何恢复？

**U4. 密钥与敏感数据**
> 系统中的密钥、Token、密码在哪里存储、如何传输、是否会出现在日志中？有哪些场景可能导致密钥泄露？

**U5. 外部依赖不可用**
> 如果 [第三方服务/API] 下线30分钟，系统的哪些功能会完全不可用？是否有降级策略？用户会看到什么？

**U6. 数据丢失场景**
> 系统在什么情况下可能丢失用户数据？数据库宕机、服务器重启、部署回滚分别会导致哪些数据丢失风险？

**U7. 错误状态恢复**
> 如果系统进入了一个"错误状态"（如某条数据卡在"处理中"状态），运维人员如何发现并修复它？是否有管理后台或恢复工具？

**U8. 空状态处理**
> 系统中所有列表类接口在返回空结果时的行为是否被明确定义？前端会正确处理空数组还是会崩溃？

**U9. 极值输入**
> 用户提交了一个包含 100MB 内容的请求、或者一个字段包含 10000 个字符、或者上传了一个 0 字节的文件，系统会如何响应？

**U10. 线上问题排查**
> 用户反馈"某个操作偶尔失败"，但没有报错。运维工程师需要多少时间，通过什么手段才能找到原因？当前设计的日志、链路追踪、监控告警是否足够支撑这个排查过程？

## 二、Web/API 附加层（Web 应用和 API 服务追加，5 题）

这些问题针对有 HTTP 接口、用户认证、浏览器交互的系统。

**W1. 峰值并发**
> 如果 [系统核心操作] 同时被 1000 个用户触发，系统会在哪个环节首先崩溃？数据库连接池、消息队列、还是某个单点服务？

**W2. 水平越权**
> 用户 A 能否通过修改请求中的 ID 参数来访问用户 B 的私有数据？现有的权限检查在哪一层实现？

**W3. 认证绕过**
> 哪些 API 端点没有鉴权？这些端点的公开是有意为之的吗？是否存在通过未授权端点间接影响系统状态的路径？

**W4. 输入注入**
> 用户输入的内容最终会以什么形式出现在：SQL 查询、Shell 命令、HTML 渲染、第三方 API 调用中？每个路径是否都有相应的过滤/转义？

**W5. 接口扩展性**
> 6个月后，[核心接口] 需要新增3个字段，现有客户端会如何受影响？这个接口是否有版本控制策略？

---

## 三、数据密集型附加层（数据密集型系统追加，5 题）

这些问题针对涉及大量数据存储、查询、迁移、缓存的系统。

**D1. 慢查询定时炸弹**
> 当数据库中的 [核心表] 数据量增长到 1000 万条时，哪些接口的查询会开始明显变慢？现有索引设计能支撑吗？

**D2. 缓存一致性**
> 如果使用了缓存，数据被直接修改数据库后，缓存什么时候失效？在失效前，用户会读到脏数据吗？

**D3. 数据库 Schema 变更**
> 如果需要对 [核心表] 进行非空字段添加或字段类型变更，能否在不停机的情况下完成？

**D4. 新增服务方**
> 如果3个月后需要接入第二个外部数据源/服务方，当前的架构需要改动哪些地方？改动范围是局部的还是全局的？

**D5. 时区与时间**
> 系统中涉及时间的逻辑（到期判断、排序、展示）在不同时区的用户那里是否行为一致？数据库存储的是 UTC 还是本地时间？
